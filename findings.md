# 调研与发现

## 已确认需求

- 参考 PDF 也是后续要标注的数据，不要求当前读取 PDF 内容。
- 需要面向专家的 Web 前端。
- 每条标注任务会提供对应标签和审查要点。
- 专家根据原文核对结果是否正确。
- 判断错误时，专家需要修改并提交正确内容。
- 用户已同意使用浏览器可视化伴侣展示布局草图、流程图和方案对比。
- 首版保持简单，仅建设专家标注端；不做管理员后台、任务分配、多人复核、冲突仲裁和复杂统计。
- 首版通过网页文件选择器导入与 `参考.json` 结构类似的 JSON，在浏览器本地解析并生成标注任务；不建设数据管理后台。
- 专家可修改判断状态、证据原文、证据页码和判断依据；证据片段支持新增、编辑和删除。
- 标签与审查要点作为固定、只读的核对依据，专家不可修改。
- 完成标注后下载一份新的 UTF-8 JSON 结果文件，原始导入文件保持不变。
- 专家同时选择对应 PDF；页面内嵌显示 PDF，点击证据页码即可跳转至相应页面核对。应用只在专家浏览器中呈现 PDF，不预先读取当前工作区 PDF 内容。

## 工作区现状

- 当前目录只有 `BK20250068.pdf`、`参考.json` 和本次建立的三份规划记录，没有已有 Web 项目或原型可复用。
- 按用户要求，没有读取 PDF 内容。
- `参考.json` 的顶层为 `output` 数组；单条记录可见字段包括 `review_point`、`verification_status`、`evidence_fragments`、`judgment_basis`、`page_numbers`。
- `evidence_fragments` 中可见 `page_number`、`original_text`、`evidence_role`，这可以映射为专家端的页码定位、原文证据和证据类型。
- 直接读取 `参考.json` 时终端显示乱码，说明后续正式导入需要显式按 UTF-8 处理并做编码验证。
- `.git` 目录存在但为空，Git 命令报告当前不是仓库；不能查看最近提交，也不能在设计阶段按技能默认流程提交规格，除非用户之后授权初始化仓库。

## 设计假设

- 以导入 JSON 的每个 `output` 数组元素作为一条独立标注任务；证据片段是任务内的可编辑子项。
- 用户选定首版为无需账号、数据库或服务端存储的本地纯前端应用；后续视情况扩展后台能力。
- 推荐技术形态为 React + TypeScript + PDF.js，业务数据通过浏览器内存/本地草稿管理，并通过 JSON 文件导出。

## 已确认设计

- 用户批准架构与数据流：文件导入、原始/工作副本分离、PDF 查看器、标注编辑器、浏览器本地草稿、UTF-8 JSON 导出。
- 未知 JSON 字段原样保留；`page_numbers` 从证据页码自动整理；草稿不持久化 PDF 文件本身。
- 用户选择 A“三栏连续工作台”：左侧持续显示任务列表，中间显示 PDF，右侧显示只读审查依据与可编辑标注表单。
- 终端回复是本次布局选择的权威反馈；浏览器事件文件未生成，不影响选择结果。
- 用户批准页面交互、校验、异常处理、部分/完整导出和测试策略；完整设计已形成书面规格。
- 规格自检明确了：可选只读 `label` 字段、判断状态固定导出格式、证据页码的内部/外部类型、物理页序号语义、任务状态派生规则，以及 IndexedDB + SHA-256 草稿隔离。
