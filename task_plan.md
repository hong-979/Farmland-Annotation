# 专家标注平台工作计划

## 目标

设计并最终实现一个 Web 专家标注平台：系统向专家提供标签、审查要点和待核对原文，专家判断内容是否正确；若错误，则修改并提交标注结果。参考 PDF 本身也是后续待标注数据，本阶段不读取其内容。

## 当前阶段

### 阶段 1：需求与设计（已完成）

- 检查项目现状、文档和已有资产。
- 逐项澄清用户角色、数据导入、标注流程和交付边界。
- 比较可行技术/交互方案。
- 形成经用户确认的设计规格。

当前状态：设计规格已写入、通过自检并获得用户批准。

### 阶段 2：实施计划（已完成）

- 仅在用户批准书面设计规格后创建详细实施计划。

当前状态：详细实施计划已写入并通过规格覆盖、占位符和类型一致性自检。

### 阶段 3：实现与验证（进行中）

- 按批准的实施计划开发。
- 验证专家端核心流程和数据持久化。

当前状态：已在隔离工作树 `expert-annotation-platform` 上完成 Task 1（脚手架）、Task 2（领域契约与合成夹具）、Task 3（JSON 解析与标准化）、Task 4（任务校验与任务列表派生状态）、Task 5（不可变编辑和安全 JSON 导出）、Task 6（SHA-256 文件指纹和 IndexedDB 草稿隔离）与 Task 7（本地 JSON/PDF 导入和可操作错误反馈），并通过子代理实现、规格审查、代码质量审查和本地验证闭环。

下一步：执行 Task 8，实现三栏工作台与任务导航。

## 关键约束

- 不读取参考 PDF 内容。

## 错误记录

| 错误 | 尝试 | 处理 |
|---|---:|---|
| Git 报告“not a git repository” | 1 | 检查发现 `.git` 为空；记录环境现状，不擅自初始化仓库。 |
| `参考.json` 直接输出为乱码 | 1 | 仅记录数据结构；后续导入设计要求显式 UTF-8 解码与校验。 |
| `planning-with-files` 的 `session-catchup.py` 不存在 | 1 | 已确认该技能安装目录只有 `SKILL.md`；改为直接读取三份规划文件恢复上下文。 |
| 内置浏览器控制接口未暴露 | 1 | 改用可视化伴侣自带本地服务，并向用户提供本地 URL。 |
| `bash` 指向未安装的 WSL 环境 | 1 | 不使用 shell 启动脚本，改为检查并直接运行其 Node 服务入口。 |
| 可视化 Node 服务通过 `Start-Process` 后立即退出 | 1 | 前台诊断证明服务本身可启动；改用由工具单元保持的前台长进程托管。 |
| 可视化服务前台诊断达到预设超时 | 1 | 这是用于捕获启动信息的预期超时，已确认服务输出正常。 |
| 占位符扫描首次因 `rg` 无匹配返回退出码 1 | 1 | 将“无匹配”显式视为成功结果并重新执行，确认文档无占位符。 |
| 实施计划自检补丁上下文不匹配 | 1 | 补丁安全失败且未改文件；读取准确段落后拆成较小定点补丁。 |
| 并行执行 `git add` 与 `git commit` 触发 worktree `index.lock` 竞争 | 1 | 确认锁文件已自行释放后改为串行提交；未做手工删除。 |
