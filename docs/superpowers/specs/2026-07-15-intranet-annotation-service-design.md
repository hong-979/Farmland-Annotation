# 内网标注服务版设计规格

- 日期：2026-07-15
- 状态：用户已审阅并批准
- 版本定位：最小可用内网服务版
- 目标：在保留现有标注工作台体验的前提下，为多人标注提供管理员上传、账号管理、任务领取、历史记录与服务端导出能力

## 1. 背景与目标

当前首版系统已经完成本地前端标注闭环：专家可导入本地 JSON 与 PDF，在同一界面中查看 PDF、修改判断与证据，并导出 UTF-8 JSON。

本次扩展不追求做成完整通用平台，而是将现有系统升级为一个可在局域网或专网内部署、可供多名标注员共同使用的最小服务版。该版本优先满足以下目标：

- 管理员可以创建标注员账号。
- 管理员可以逐条上传一份 `JSON + PDF`。
- 系统可将上传 JSON 中的 `output[]` 拆成任务池。
- 标注员登录后可自行领取下一条待处理任务。
- 标注过程继续使用当前的 PDF + 标注工作台交互。
- 系统在服务端保存草稿与提交结果，并记录操作历史。
- 管理员可导出文档级 JSON 结果。
- 系统可部署在一台内网机器上，由同网络中的其他用户访问。

## 2. 范围

### 2.1 包含

- 登录鉴权，角色仅分为管理员与标注员。
- 管理员创建账号、停用账号。
- 管理员单条上传一份 `JSON + PDF`。
- 服务端拆分任务并建立任务池。
- 标注员领取下一条待处理任务。
- 标注员继续编辑自己已领取但未提交的任务。
- 服务端自动保存草稿。
- 标注员提交任务。
- 管理员可回收异常卡住的已领取任务，使其重新回到任务池。
- 管理员查看文档级任务状态与单任务历史。
- 管理员执行部分导出与完整导出 JSON。
- 系统以 Web 服务方式运行，允许同网络其他终端访问。

### 2.2 明确砍掉

- 批量上传。
- 标注员注册、找回密码、验证码登录。
- 管理员手动分配任务。
- 标注员浏览全部任务池。
- 多人同时编辑同一任务。
- 字段级 diff 审计。
- 审核、复核、驳回、仲裁流程。
- 图表统计与复杂报表。
- Word、Excel、PDF 等其他格式导出。
- 公网 SaaS 化、多租户、复杂权限体系。
- Docker 必选化、对象存储、反向代理高可用方案。

## 3. 推荐方案与取舍

本版采用单体内网服务方案：

- 前端：继续使用现有 React + TypeScript 标注界面。
- 后端：新增 Node.js 服务，提供认证、上传、任务、历史、导出接口。
- 数据库：使用 SQLite 保存结构化数据。
- 文件存储：使用服务器本地磁盘保存上传源文件。

选择该方案的原因：

- 改造量最小，能最大程度复用现有前端与 JSON 处理逻辑。
- 内网部署简单，不需要额外数据库服务。
- 后续若要升级到更正式架构，仍可保留前端与领域模型。

不采用以下方案：

- 前后端大规模分离并引入 PostgreSQL：首版实现速度会明显下降。
- 只补少量接口继续依赖浏览器本地草稿：多人协作边界不清晰，服务化价值不足。

## 4. 系统边界

系统拆分为四个明确单元：

1. `标注前端`
   复用现有三栏工作台与 JSON 标注编辑逻辑，通过 API 获取任务、保存草稿、提交结果。

2. `管理后台`
   提供最小管理员能力：账号管理、单条上传、文档列表、文档详情、任务历史、回收任务、导出。

3. `应用后端`
   负责登录鉴权、文件上传、任务拆分、任务领取、草稿保存、提交、历史记录、PDF 受控访问与 JSON 导出。

4. `数据存储`
   SQLite 保存用户、文档、任务、历史；磁盘目录保存原始 `JSON` 与原始 `PDF`。

## 5. 数据模型

为了加快实现，本版只保留五类核心数据。

### 5.1 `users`

保存账号信息。

建议字段：

- `id`
- `username`
- `password_hash`
- `role`，取值为 `admin` 或 `annotator`
- `display_name`
- `status`，取值为 `active` 或 `disabled`
- `created_at`

### 5.2 `documents`

表示管理员上传的一份文档包，即一组 `JSON + PDF`。

建议字段：

- `id`
- `title`
- `json_file_path`
- `pdf_file_path`
- `source_json_name`
- `source_pdf_name`
- `task_count`
- `created_by`
- `created_at`

### 5.3 `tasks`

表示从 `output[]` 中拆出的独立任务。

建议字段：

- `id`
- `document_id`
- `task_index`
- `label`
- `review_point`
- `original_payload_json`
- `current_payload_json`
- `status`
- `claimed_by`
- `claimed_at`
- `submitted_by`
- `submitted_at`
- `updated_at`

关键约束：

- `original_payload_json` 永不修改。
- `current_payload_json` 始终保存当前最新可导出结构。
- 首版不把证据片段拆成独立子表，避免数据库模型膨胀。

### 5.4 `task_history`

保存任务历史记录。

建议字段：

- `id`
- `task_id`
- `actor_user_id`
- `action_type`
- `snapshot_json`
- `created_at`

`action_type` 首版只保留：

- `claim`
- `reclaim`
- `submit`

历史记录按整条任务快照保存，不做字段级比对。

## 6. 任务状态

任务状态仅保留三种：

- `pending`：尚未被领取。
- `claimed`：已被某位标注员领取，尚未提交。
- `submitted`：标注员已提交。

首版不区分更多中间态，以减少判断分支与界面复杂度。

## 7. 核心流程

### 7.1 管理员上传

管理员在后台填写文档标题并上传一份 `JSON + PDF`。

后端执行：

- 校验 JSON 是否为 UTF-8 且顶层包含 `output[]`。
- `output[]` 中每一项必须至少包含非空 `review_point`，`label` 为可选字段。
- 将源文件保存到服务器磁盘。
- 将 `output[]` 拆成任务并写入 `tasks` 表。
- 初始化任务状态为 `pending`。

首版不做批量上传、自动配对、复杂预览编排。

### 7.2 标注员领取任务

标注员登录后，只有两个主入口：

- `领取下一条任务`
- `继续我已领取任务`

点击“领取下一条任务”时，后端从 `pending` 任务中原子领取一条，立即写入：

- `status = claimed`
- `claimed_by = 当前标注员`
- `claimed_at = 当前时间`

这样可避免两名标注员同时领取同一条任务。

为保持前后端最简单一致，首版明确约束：

- 每名标注员同一时刻最多只能持有一条 `claimed` 且未提交的任务。
- 若该标注员已有未提交任务，`claim-next` 返回冲突提示，并要求其继续当前任务。
- `GET /api/annotator/tasks/current` 只返回该标注员当前唯一未提交任务；若不存在则返回空。

管理员在异常场景下保留一项兜底能力：

- 可将某条 `claimed` 任务回收为 `pending`，用于处理标注员离岗、忘记提交或无法继续处理的情况。
- 回收时清空 `claimed_by` 与 `claimed_at`，保留该任务当前最新草稿 `current_payload_json`。
- 回收动作必须写入 `task_history(action_type = reclaim)`。

### 7.3 标注与草稿保存

标注页面继续复用现有三栏结构与右侧编辑工作台：

- 中间查看 PDF。
- 右侧修改 `verification_status`、`evidence_fragments`、`judgment_basis`。
- 证据页码支持跳转 PDF。

保存策略改为服务端为准：

- 前端维持当前即时编辑体验。
- 经过短延迟自动调用保存草稿接口。
- 后端将整条任务当前 JSON 写入 `current_payload_json`。
- 草稿保存不要求每次都新增历史记录；首版只要求覆盖最新草稿，并记录 `updated_at`。
- 历史记录只记录显式关键动作，首版仅要求 `claim` 与 `submit` 可追溯。
- `draft` 接口只接受当前用户对自己 `claimed` 任务的保存请求；若任务已变为 `submitted` 或已被回收，则直接拒绝。
- 标注员点击提交时，前端必须先停止未触发的自动保存定时器；提交成功后，前端进入只读状态，并忽略所有更早发出的草稿保存响应。

本版不再依赖浏览器 `IndexedDB` 作为主草稿存储。

### 7.4 提交任务

标注员点击提交后，后端执行任务校验。

首版沿用当前规则：

- 必须选择判断状态。
- 如果判断为“错误”，至少一条证据且判断依据非空。
- 证据页码必须为正整数。

校验通过后：

- `status` 改为 `submitted`
- 写入 `submitted_by` 与 `submitted_at`
- 追加一条 `task_history(action_type = submit)`

首版不支持标注员自行撤回提交。

### 7.5 管理员导出

管理员可对单份文档执行：

- 部分导出 `partial`
- 完整导出 `final`

导出时，后端按任务顺序取出 `current_payload_json`，以原始整份 JSON 为基底，仅替换其中的 `output[]`，保留所有未知字段，并直接返回下载响应。

导出语义明确如下：

- 上传拆分任务时，`current_payload_json` 先初始化为 `original_payload_json`。
- `partial` 导出允许文档中存在未提交任务。
- `partial` 导出时：
  - `submitted` 任务使用已提交的 `current_payload_json`。
  - `claimed` 任务使用当前最新草稿 `current_payload_json`。
  - `pending` 任务使用初始化后的 `current_payload_json`，即原始任务内容。
- `final` 导出要求该文档下全部任务均为 `submitted`；只要存在 `pending` 或 `claimed` 任务，就拒绝导出并返回中文提示。
- 首版不再区分其他导出中间态。
- 首版不持久化导出文件，不记录导出历史表，不做导出文件清理策略。

## 8. 页面边界

首版前端页面仅保留四个。

### 8.1 登录页

管理员与标注员共用。

包含：

- 用户名输入
- 密码输入
- 登录提交

不包含：

- 注册
- 找回密码
- 验证码

### 8.2 标注员工作台

在现有标注工作台基础上做服务化改造。

保留：

- 三栏布局
- PDF 查看
- 判断编辑
- 证据编辑
- 判断依据编辑

改动：

- 移除本地导入入口
- 增加任务领取与继续编辑入口
- 顶部显示当前任务所属文档信息
- 自动保存改为服务端接口
- 提交改为服务端接口
- PDF 通过登录态接口拉取完整二进制，再转换为浏览器本地 `blob URL` 交给现有 PDF 查看器
- 本地导出入口从标注员界面移除

### 8.3 管理员后台首页

使用极简表格展示文档列表。

管理员首页同时承接两项必需管理能力：

- 页面顶部放置单条上传表单。
- 页面侧边或弹窗中提供账号管理区，用于创建与停用标注员账号。

这两项均不新增独立页面，避免导航与实现复杂度继续膨胀。

每条文档至少显示：

- 文档标题
- 上传时间
- 总任务数
- `pending / claimed / submitted` 数量
- 导出入口
- 详情入口

不做图表与复杂筛选。

### 8.4 管理员文档详情页

包含以下能力：

- 查看任务状态列表
- 查看某条任务历史记录
- 回收异常卡住的 `claimed` 任务
- 执行导出

首版不允许管理员在该页面直接在线标注。

## 9. 接口边界

后端 API 只保留三组。

### 9.1 认证接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

首版推荐使用基于 cookie 的 session。

### 9.2 标注员接口

- `POST /api/annotator/tasks/claim-next`
- `GET /api/annotator/tasks/current`
- `GET /api/annotator/tasks/:taskId`
- `PUT /api/annotator/tasks/:taskId/draft`
- `POST /api/annotator/tasks/:taskId/submit`

首版不开放“浏览全部任务池”接口。

### 9.3 管理员接口

- `POST /api/admin/users`
- `GET /api/admin/users`
- `PATCH /api/admin/users/:id/status`
- `POST /api/admin/documents`
- `GET /api/admin/documents`
- `GET /api/admin/documents/:id`
- `GET /api/admin/documents/:id/tasks`
- `GET /api/admin/tasks/:taskId/history`
- `POST /api/admin/tasks/:taskId/reclaim`
- `POST /api/admin/documents/:id/export`
- `GET /api/documents/:id/pdf`

`GET /api/documents/:id/pdf` 的契约固定如下：

- 返回完整 PDF 文件二进制。
- 首版只要求受登录态保护的普通下载接口，不要求 `Range`、`HEAD` 或流式分块能力。
- 前端收到响应后自行创建 `blob URL` 供现有 PDF 查看器使用。

## 10. 并发与权限规则

首版只实现最必要的控制：

- `pending -> claimed` 必须通过原子更新完成，保证同一任务只会被一人领取成功。
- `claimed` 状态下，仅领取人本人可修改任务；管理员仅可查看任务与历史，不可直接修改标注内容。
- `submitted` 状态下，标注员只读；管理员仅可查看历史与导出。
- 文档级 PDF 读取权限规则如下：
  - 管理员可读取任意文档 PDF。
  - 标注员仅可读取自己当前 `claimed` 任务所属文档的 PDF。
- 停用账号无法登录，也无法继续操作既有任务。
- 若标注员仍持有 `claimed` 且未提交任务，则管理员不能直接停用该账号；需先回收该任务，再执行停用。

首版不做协同编辑、乐观锁冲突合并、字段级权限。

## 11. 文件与部署

### 11.1 文件存储布局

建议服务端目录结构：

- `data/app.db`
- `data/uploads/json`
- `data/uploads/pdf`

数据库仅记录相对路径。

### 11.2 运行方式

首版按“前后端一体运行”的方式交付：

- 前端构建产物由后端静态托管。
- 后端同时提供页面与 API。
- 服务监听固定端口，允许同网络终端通过 `http://服务器IP:端口` 访问。

### 11.3 打包交付

交付物最小化为：

- `server`
- `web`
- `data`
- `start.bat`
- `stop.bat`

部署机器只要求安装 Node.js，并放通服务端口。

### 11.4 首个管理员初始化

由于只有管理员可以创建账号，首版必须提供首个管理员初始化方式。

采用最简单方式：

- 服务端首次启动时读取环境变量中的默认管理员账号与密码。
- 若数据库中不存在管理员账号，则自动创建首个管理员。
- 若数据库中已存在管理员，则忽略该初始化逻辑。

这样可避免额外制作初始化页面或安装向导。

## 12. 错误处理

系统错误提示统一使用中文，并说明三件事：

- 发生了什么。
- 哪一条数据或哪一个步骤受影响。
- 用户下一步可以怎么做。

重点覆盖：

- 登录失败
- 停用仍持有未提交任务账号失败
- 回收任务失败
- 上传文件结构错误
- JSON 解析失败
- PDF 无法读取
- 任务领取失败
- 草稿保存失败
- 提交校验失败
- 导出失败

## 13. 测试与验收

### 13.1 必测链路

- 管理员登录成功。
- 管理员创建标注员账号。
- 管理员上传单条 `JSON + PDF`。
- 上传后任务被正确拆分。
- 标注员登录并领取下一条任务。
- 标注员保存草稿并再次打开继续编辑。
- 标注员提交任务。
- 管理员查看任务历史。
- 标注员已有未提交任务时无法重复领取下一条任务。
- 管理员可回收异常卡住的 `claimed` 任务，回收后其他标注员可重新领取。
- 管理员无法停用仍持有未提交任务的标注员账号。
- 管理员导出 JSON。
- 前端可以通过受控接口读取并展示 PDF。
- 同网络其他机器通过 `IP:端口` 正常访问。

### 13.2 不作为首版阻塞项

- 高并发压测。
- 海量文件性能优化。
- 复杂安全攻防。
- 监控告警体系。
- 全量端到端自动化覆盖。

### 13.3 验收标准

满足以下条件即可视为本版完成：

1. 管理员可创建并停用标注员账号。
2. 管理员可逐条上传 `JSON + PDF` 并生成任务池。
3. 标注员可自行领取下一条任务并继续编辑未提交任务。
4. 标注页面保留当前 PDF 与证据编辑体验。
5. 草稿与提交结果保存于服务端，而非仅保存在浏览器本地。
6. 至少每次领取、回收与提交都有历史记录可查，草稿始终以最新服务端版本为准。
7. 管理员可导出部分或完整 UTF-8 JSON。
8. 系统可在一台内网主机上运行，并允许其他网络终端访问。

## 14. 后续扩展边界

后续如需扩展，可在当前基础上追加：

- 管理员手动分配任务
- 批量上传
- 审核复核流
- 更丰富的统计看板
- 更正式的数据库与部署架构

但这些扩展不应反向影响本版“最小可用、最快落地”的实现边界。
