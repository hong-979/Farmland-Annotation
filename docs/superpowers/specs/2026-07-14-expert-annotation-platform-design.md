# 专家标注平台设计规格

- 日期：2026-07-14
- 状态：用户已审阅并批准
- 首版形态：本地纯前端 Web 应用
- 适用对象：对高标准农田初步设计报告进行核对与修订的专家

## 1. 背景与目标

首版建设一个本地运行的专家标注 Web 应用。专家选择一份结构类似 `参考.json` 的标注数据和对应 PDF，在同一页面中查看审查任务与 PDF 原文，核对已有判断是否正确，并修改判断状态、证据和判断依据。完成后，应用下载一份新的 UTF-8 JSON 结果文件，原始 JSON 和 PDF 均保持不变。

当前 PDF 只是后续待标注数据。本设计与实施前置分析不读取该 PDF 内容；实际应用仅在专家自己的浏览器中加载和显示专家选择的 PDF。

## 2. 首版范围

### 2.1 包含

- 在网页中选择本地 JSON 和对应 PDF。
- 按 JSON 的 `output` 数组生成任务列表。
- 内嵌显示 PDF，支持翻页、页码输入、缩放和适应宽度。
- 点击证据页码跳转到 PDF 对应页面。
- 只读展示标签和审查要点。
- 编辑判断状态、证据片段和判断依据。
- 新增、编辑和删除证据片段。
- 从证据页码自动生成任务的 `page_numbers`。
- 在浏览器本地自动保存标注草稿。
- 导出完整结果或带 `partial` 标记的部分结果 JSON。

### 2.2 不包含

- 用户账号、登录和权限系统。
- 管理员后台、任务分配和数据管理页面。
- 服务端文件上传、数据库和远程存储。
- 多专家协作、复核、冲突仲裁和统计报表。
- PDF 内容自动抽取、OCR、自动审查或模型推理。

这些能力可以在后续版本中通过既定接口扩展，但首版不为尚未确认的需求增加界面和业务复杂度。

## 3. 技术形态

应用采用 React、TypeScript 和 PDF.js 构建，通过本地开发服务器运行，也可以打包为静态站点。首版不依赖后端服务。

系统分为五个边界明确的模块：

1. `FileImport`：读取用户选择的 JSON，创建 PDF 本地对象地址，并执行输入校验。
2. `TaskRepository`：维护不可变原始快照和可编辑工作副本，提供任务查询、更新和完成状态计算。
3. `PdfViewer`：封装 PDF.js，响应页码跳转和缩放操作，不接触标注业务规则。
4. `AnnotationEditor`：展示只读审查依据，编辑判断、证据和判断依据，执行单条任务校验。
5. `DraftAndExport`：按文件内容指纹保存/恢复浏览器草稿，将工作副本安全合并到原 JSON 并导出。

草稿和导出接口必须与具体存储实现解耦。后续若增加服务器，只替换存储/同步适配器，不重写任务模型、PDF 查看器和标注编辑器。

## 4. 数据模型与字段权限

JSON 顶层必须包含 `output` 数组。每个数组元素是一条独立标注任务，`evidence_fragments` 是该任务的证据子项。

| 字段 | 首版用途 | 专家权限 | 导出规则 |
|---|---|---|---|
| `label`（可选） | 上游提供的独立标签 | 只读 | 存在时原样保留 |
| `review_point` | 审查要点 | 只读 | 原样保留 |
| `verification_status` | 专家判断 | 可编辑 | 固定导出为 `[正确]`、`[错误]` 或 `[未涉及]` |
| `evidence_fragments` | 证据列表 | 可新增、编辑、删除 | 保留列表顺序并导出最终内容 |
| `evidence_fragments[].page_number` | 证据页码与 PDF 跳转 | 可编辑 | 内部按正整数处理，导出为十进制数字字符串 |
| `evidence_fragments[].original_text` | 专家核对的证据原文 | 可编辑 | 导出最终文本 |
| `evidence_fragments[].evidence_role` | 证据角色 | 可编辑 | 导出最终值 |
| `judgment_basis` | 判断依据 | 可编辑 | 导出最终文本 |
| `page_numbers` | 任务涉及页码汇总 | 不直接编辑 | 从证据页码去重并升序生成整数数组 |
| 其他未知字段 | 上游扩展信息 | 不展示、不编辑 | 深拷贝并原样保留 |

应用必须同时保留：

- 原始快照：导入后不可修改，用于比较、恢复和安全合并。
- 工作副本：承载专家全部修改。

任何编辑都只能作用于工作副本，不得原地修改原始快照。

判断状态在应用内部使用 `correct`、`incorrect`、`not_applicable` 三个枚举值，界面显示“正确”“错误”“未涉及”，导出时映射为与参考数据一致的带方括号中文值。若 JSON 不包含独立 `label` 字段，界面只显示 `review_point`，不得从审查要点文本中猜测或拆分标签。

首版将证据页码解释为 PDF 文件的物理页序号，从 1 开始；不处理封面导致的印刷页码偏移，也不提供页码偏移配置。

## 5. 页面结构

用户选择了 A“三栏连续工作台”。可视化参考位于 `.superpowers/brainstorm/layout-20260714/content/expert-layout-options.html`。

### 5.1 顶部工具栏

- 显示当前 JSON 和 PDF 文件名。
- 显示已处理任务数、任务总数和整体进度。
- 提供“重新选择文件”“导出当前草稿”和“完成并导出”入口。

### 5.2 左侧任务栏

- 持续展示全部任务，方便批量连续处理。
- 每条任务显示序号、简短审查要点和状态标记。
- 状态包括未处理、已确认、已修改和信息不完整。
- 点击任务时，中间 PDF 和右侧编辑器切换到对应任务。

### 5.3 中间 PDF 区

- 显示当前 PDF 页面。
- 提供上一页、下一页、页码输入、缩放和适应宽度。
- 接收右侧证据页码发出的跳转指令。
- PDF 无法使用时保留占位与错误提示，不阻断 JSON 任务查看。

### 5.4 右侧标注区

按以下顺序排列：

1. 只读标签和审查要点。
2. 专家判断：正确、错误、未涉及。
3. 证据片段列表及新增/删除操作。
4. 判断依据文本编辑区。
5. 草稿状态和“保存并下一条”。

在小尺寸窗口下，左侧任务栏允许折叠；PDF 与编辑区仍保持双栏。首版以桌面端使用为主，不要求手机端完成复杂标注。

## 6. 用户流程

1. 打开应用，选择 JSON 文件。
2. 应用按 UTF-8 解析并校验 JSON；校验成功后生成任务列表。
3. 选择对应 PDF；PDF 在浏览器内建立本地对象地址并加载。
4. 进入三栏工作台，默认打开第一条未处理任务。
5. 查看只读审查要点，通过证据页码跳转 PDF 并核对原文。
6. 选择或修正判断状态，编辑证据和判断依据。
7. 保存并进入下一条；每次编辑同时写入本地草稿。
8. 可随时导出带 `partial` 标记的部分结果。
9. 全部任务通过校验后完成并导出新的 JSON。

## 7. 草稿策略

- 草稿按 JSON 文件内容指纹隔离，不能仅按文件名匹配。
- 重新选择内容相同的 JSON 时提示恢复草稿。
- 同名但内容不同的 JSON 不应用旧草稿。
- 草稿保存工作副本、当前任务索引和必要的界面状态。
- PDF 文件及其内容不写入草稿；恢复后由专家重新选择 PDF。
- 导出成功后不自动删除草稿，先提示专家确认；首版可提供显式“清除本地草稿”操作。

草稿使用浏览器 IndexedDB 保存；JSON 内容指纹使用文件字节的 SHA-256。PDF 文件、PDF 字节和 PDF 文本不得写入 IndexedDB。

## 8. 校验规则

### 8.1 导入校验

- 文件必须能按 UTF-8 解析为 JSON。
- 顶层必须存在数组类型的 `output`。
- 每条任务必须包含非空的 `review_point`；`label` 是可选只读字段。
- `evidence_fragments` 缺失时规范化为空数组；类型错误时报告具体任务序号。
- 可选字段缺失时补为安全空值并给出非阻塞提示。

### 8.2 任务校验

- 必须选择专家判断。
- 判断为“错误”时，至少包含一条证据且判断依据不能为空。
- 证据页码必须是正整数。
- PDF 已加载时，证据页码不得大于 PDF 总页数。
- 超界或格式错误直接标记在对应证据卡片，不允许该任务进入完成状态。
- `page_numbers` 始终由证据页码计算，界面不提供独立输入框。

任务列表状态按以下规则派生：未选择专家判断时为“未处理”；任务存在校验错误时为“信息不完整”；工作副本与原始快照的可编辑字段不同且校验通过时为“已修改”；其余已选择判断且校验通过的任务为“已确认”。

### 8.3 导出校验

- “导出当前草稿”允许存在未完成任务，输出文件名包含 `partial`。
- “完成并导出”要求全部任务通过任务校验。
- 生成下载文件前，应用必须重新解析序列化结果，确认其为有效 JSON。
- 导出编码固定为 UTF-8，原文件不被覆盖。

## 9. 文件命名

完整结果：

`原文件名_专家标注_YYYYMMDD-HHmm.json`

部分结果：

`原文件名_专家标注_partial_YYYYMMDD-HHmm.json`

## 10. 错误处理

- JSON 语法错误：显示可定位的解析错误，不进入工作台。
- JSON 结构错误：指出任务序号、字段和期望类型。
- PDF 加载失败：保留 JSON 标注能力，禁用页码跳转并提示重新选择 PDF。
- 页码越界：在证据项内就地提示，任务状态标为信息不完整。
- 草稿读写失败：保留当前内存数据，持续显示未持久化警告，并允许立即导出部分结果。
- 导出失败：不修改当前工作副本，显示可重试错误。
- 页面离开：存在未导出改动时显示浏览器离开提醒。

错误消息使用中文，说明“发生了什么、哪一条数据受影响、专家可以怎么处理”，不只显示技术异常文本。

## 11. 测试策略

### 11.1 单元测试

- JSON 解析和字段规范化。
- 原始快照与工作副本隔离。
- 证据新增、编辑和删除。
- `page_numbers` 去重与排序。
- 文件内容指纹、草稿隔离和恢复。
- 未知字段保留及导出合并。
- 完整/部分结果文件名生成。

### 11.2 组件测试

- 任务列表状态与切换。
- 只读字段不可编辑。
- 判断、证据和判断依据编辑。
- 页码点击向 PDF 查看器发送正确跳转指令。
- 字段错误在对应控件附近显示。

### 11.3 端到端测试

- 导入有效 JSON 和测试 PDF，完成一条标注并导出。
- 新增、修改、删除证据后导出结果正确。
- 刷新页面后恢复同一 JSON 的草稿。
- 同名但内容不同的 JSON 不错误恢复草稿。
- PDF 加载失败时仍可导出 JSON 草稿。
- 未完成任务只能导出带 `partial` 标记的结果。
- 全部任务通过校验后导出完整结果。

测试只使用人工构造的小型 JSON 和空白测试 PDF，不读取当前提供的业务 PDF 内容。

## 12. 验收标准

首版满足以下条件时可以验收：

1. 专家能在本地页面选择 JSON 和 PDF，文件不上传到服务端。
2. JSON 的每个 `output` 元素生成一条可导航任务。
3. 标签和审查要点只读且无法通过界面修改。
4. 判断状态、证据页码、证据原文、证据角色和判断依据均可编辑。
5. 证据支持新增和删除，点击页码能跳转到 PDF 对应页。
6. 修改内容自动保存为按文件内容隔离的本地草稿。
7. 可导出部分和完整 UTF-8 JSON，原始文件不变且未知字段不丢失。
8. 输入、页码、草稿和导出异常都有可操作的中文提示。
9. 自动测试覆盖核心数据规则和完整标注流程。

## 13. 后续扩展边界

后续如需开放后台，可新增远程存储、账号、任务分配和多人协作适配器。首版的领域模型和编辑组件不直接依赖浏览器文件 API 或本地草稿实现，确保未来扩展时复用当前前端，而不是重新设计专家标注流程。
